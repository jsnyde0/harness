# ADR-002: Cross-Harness Substrate — Topology and Thin-Waist Architecture

Date: 2026-06-03
Originated from: brainstorm session "claude-code-for-pi-agents"

## Status

Proposed — **all decisions EXPLORATORY unless noted.** This is an active experiment expected to evolve in place (ADR-011-style) as the pi-primary workflow is dogfooded. Do not treat any decision here as binding architecture; the firmness labels say "this will move." **(D2 + D3 revised 2026-06-04** with empirical findings from a Claude Code schema spike + a hook-runner build + a Codex consumption-model spike (three-harness reshape: byte-identical-file → one-shared-source-projected-per-harness); **D2 further revised 2026-06-05** (Codex hook firing/trust gate) **and 2026-06-06** (instructions axis + import-pointer projection; hook event-axis projection subset); **D3 further revised 2026-06-06** (role-class field optionality); **D1 + D5 + Context revised 2026-06-04 — Codex promoted from deprioritized to a first-class harness target; new D7 adds the open-source public-core / private-overlay distribution posture**; **D7 further revised 2026-06-07** (W3 framing: operational decisions D7.1–D7.5 — canonical-home move-out, repo identity `harness` + namespace reconciliation, central-manifest/private-by-default boundary, one-way-check enforcement, uniform-faithful-descriptor scrub; **D7.6 added during the W3 scrub-bead execution** — portable-form convention for core path references; **D7.4 refined 2026-06-08** — config-driven marker externalization, so the published one-way check ships no real private nouns and is reusable by any adopter) — see those decisions.)

## Context

The trigger was an economic question: can a Claude Pro/Max **subscription** power a non-Claude-Code harness (pi) cheaply? As of **June 15, 2026**, Anthropic meters `claude -p` / Agent SDK usage against a separate, capped, API-rate-billed credit (Pro $20 … Max-20x $200) — explicitly *permitting* third-party-harness use but removing the subsidy that motivated it. For headless/programmatic work, subscription and raw API now bill at the same rate; the subsidy survives only for genuinely *interactive* use. (Interactive-emulation to tap the subsidized pool is rejected — see D1.)

That collapses the original question and surfaces the real one: **make pi the primary harness, and make the workflow substrate (instructions, skills, hooks, subagents, memory) span both Claude Code and pi — and be distributable.** Research established that the substrate splits into a *portable half* with cross-harness open standards (AGENTS.md, SKILL.md, MCP) and a *discipline half* (default-on hooks, subagent-dispatch, memory bindings) that is irreducibly harness-specific and must be re-bound per harness.

Targets are **Claude Code, pi, and Codex** (revised 2026-06-04 — Codex promoted from deprioritized to a first-class harness target; its substrate-consumption model is settled by a forthcoming spike mirroring an earlier Claude Code schema spike, see D1). pi remains the design center. Revised 2026-06-04, the substrate is additionally being extracted into a reusable open-source **public core** with a **private overlay** for personal/work content (D7). The economics that make this coherent: pi runs a cheap brain (GPT-5.5 via OpenAI subscription) + cheap execution (Kimi K2.6), while Claude Code is retained only for huge token-consuming pre-planned epics via *interactive* `/send-it` (the one place the Anthropic subsidy is legitimately available).

**Where this ADR lives, and why not the containerized-agent safety project's ADR-019:** an overlap check scored that project's ADR-019 (pi-coding-agent support) as high-overlap. That match is on surface ("a neutral config location") not substance: ADR-019 governs *how that container hosts pi* (cage-owned, image-baked config — a containerization/isolation concern). This ADR governs *cross-harness substrate portability and harness topology* (a pi-config-home concern). Different repos, different scopes. ADR-019 is referenced, not evolved, by this ADR.

## Decisions

### D1: Harness topology — pi primary, Claude Code as subsidized heavy-executor, Codex first-class (revised 2026-06-04), beads as the interface

- **Firmness:** EXPLORATORY
- **Decision:** pi is the user's primary day-to-day coding-agent harness (brain = `openai/gpt-5.5`, execution = `moonshotai/kimi-k2.6`). Claude Code is retained **only** for huge token-consuming, pre-planned epics, invoked via *interactive* `/send-it` on the legitimately-subsidized Anthropic path. The **bead substrate is the cross-harness interface**: pi plans into beads; the large ones flow to Claude Code. **No interactive-spoofing** of the Claude Code client to draw from the subsidized interactive pool (re-affirms the containerized-agent safety project's no-evasion stance) — the Anthropic subsidy is used only where work is genuinely interactive. **Revised 2026-06-04 — Codex is a first-class harness target.** The OpenAI Codex CLI (AGENTS.md-centric) joins Claude Code and pi as a substrate-consumption target; its place in the topology and its wiring (subagent mechanism? hooks? what its model field accepts? how it loads skills/instructions?) are **determined by a consumption-model spike** mirroring an earlier Claude Code schema spike — not asserted here. "First-class" means correctly bound on known facts, not parity claimed blind.
- **Rationale:** Post-June-15, headless Anthropic use is metered at API rates regardless of auth path, so there is no economic reason to route pi's bulk work through Anthropic. A cheap-brain + cheap-executor pi, with Claude Code reserved for the one subsidized-and-legitimate niche (human-launched heavy epics), minimizes spend while keeping the richest harness for the work it is best at.
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Route pi through Anthropic subscription via Agent SDK credit | "Use the sub we already pay for" | **Rejected** — external: per Anthropic help-center (support.claude.com/articles/15036540), the Agent SDK credit is capped + API-rate-billed; ~$200 credit ≈ ~10–20% of interactive-equivalent consumption. No subsidy for headless. |
  | Interactive-emulation (spoof CC client / drive it like a human) to tap the subsidized pool | The subsidy lives there | **Rejected** — external: Anthropic's Jan 2026 enforcement detects "spoofing the Claude Code harness"; risks the whole account (incl. the interactive use we want to keep). reasoned: also violates the containerized-agent safety project's no-evasion stance. |
  | Keep Claude Code as primary harness | Richest substrate already lives there | **Rejected** — reasoned: the goal is pi-primary; CC's value for routine work doesn't justify its cost/coupling once pi is viable. |
- **What would invalidate this:** pi (or GPT-5.5 as its brain) proves inadequate as a daily driver; OR OpenAI begins metering Codex/agentic subscription use the way Anthropic did (removing the cheap-brain economics); OR the bead-as-interface handoff proves too lossy in practice; OR the Codex consumption-model spike reveals Codex cannot consume the thin-waist substrate without genuine per-harness logic duplication (D2's invalidation trigger), making first-class Codex support uneconomical.

### D2: Thin-waist cross-harness architecture — shared neutral assets, per-harness wiring

- **Firmness:** EXPLORATORY (revised 2026-06-04)
- **Decision:** Substrate *logic* lives in harness-**neutral shared assets**: hook logic as standalone scripts/CLIs; subagent roles as a shared markdown role-brief library (D3); instructions as `AGENTS.md`; skills as `SKILL.md` folders. Each harness keeps its **native wiring/spawner** that *reads from* the shared source — Claude Code via `settings.json` hooks + `$CLAUDE_HOME/agents/*.md`; pi via `pi.on()` extensions + the evolved `subagent` role-loader. The harness-specific surface shrinks to thin trigger-wiring; the logic is shared.
  - **Revised 2026-06-04 (hook-runner build):** the per-harness wiring layer may also perform *declarative I/O-protocol adaptation*, and this is still "wiring," not a logic fork. Shared hook scripts have a genuinely per-harness I/O contract — Claude Code feeds the command via **stdin as JSON** and reads a `permissionDecision` field; pi's runner passes args and reads the **exit code**. The thin-waist holds because pi's generic hook-runner adapts via a **fixed, data-selected protocol enum** (`exit-code` | `dcg-json` | `cc-hook`) keyed on a manifest field: it feeds each script the input shape that script expects and interprets its block-signal accordingly, so the shared scripts run **unchanged** in both harnesses. A fixed, manifest-data-selected adapter set is wiring; adding a hook that uses an existing protocol needs zero runner code change. (Verified: the same `block-compound-commands.sh` / `dcg` scripts that Claude Code drives now fire in pi via the manifest, untouched.)
  - **Revised 2026-06-04 (Codex consumption-model spike) — the thin-waist generalizes to three harnesses; per-harness *projection* may be symlink OR compile.** The spike (evidence pinned to Codex source + live CLI) found no Codex element forces genuine per-harness logic duplication. Two refinements: (i) **role-file projection** — Codex role definitions are TOML, not markdown, so Codex cannot share the *same physical file* CC and pi share; it binds from the single shared source by **generating its role file at compile-time** — the D6 compile-skill doing its job (projection from one source, not a fork). (ii) **hooks** — Codex copied Claude Code's hook format almost verbatim, so the existing fixed protocol-adapter already covers Codex with no new logic. The wrinkle is that Codex supports only a *subset* of the `cc-hook` decision surface (`deny`+reason → block; `allow`+`updatedInput` → rewrite; exit-2+stderr → block) and **every unsupported signal FAILS OPEN — the tool proceeds, not blocks**. Unsupported includes `ask`, a `deny` emitted *without* a reason, bare `allow`, legacy `approve`, and the universal fields `continue:false`/`stopReason`/`suppressOutput`. The safety-relevant direction: a malformed-or-unsupported signal from a shared *guard* hook fails **open**, not closed — so the `cc-hook` adapter must keep guard hooks inside Codex's supported `deny`+reason subset, or the guard silently no-ops on Codex. (Source: Codex source `hooks/src/engine/output_parser.rs:362-372,434-474`; `events/pre_tool_use.rs:532` test `unsupported_permission_decision_fails_open`.) **Revised 2026-06-05 (DISCOVERY ≠ FIRING / trust gate):** beyond the decision-surface subset above, a Codex hook that parses correctly can still *never fire*. Codex reads hooks ONLY from `~/.codex/config.toml` in a `[[hooks.PreToolUse]]` array-of-tables schema (a standalone `~/.codex/hooks.toml` is never read), and a discovered hook defaults to `trustStatus: untrusted` and is **silently skipped until a `[hooks.state."<sourcePath>:pre_tool_use:<g>:<h>"]` entry carries a `trusted_hash` matching the hook's content hash**. `--dangerously-bypass-hook-trust` is *inert* for the per-thread configs that actually run tool calls (it only sets the bootstrap-config flag) — so the production mechanism that fires a shared guard hook under `codex exec` is the seeded `trusted_hash`, not the bypass flag. Operationally: wiring any shared *guard* hook onto Codex requires the D6 compile-skill to emit the correct schema **AND** seed a content-hash trust entry; a guard hook shipped without that seed is silently INERT — it enforces nothing. (Source: a Codex hook-trust build; Codex source `core/src/tools/registry.rs` `handle_tool_call_with_source` + `core/src/hook_runtime.rs` `codex.hooks.run` spans + `config/src/hooks_tests.rs` `trusted_hash` state, reconciled at commit a639e87. The trust seed is content-pinned — rewriting the hook invalidates it; see bd memory `reimplementing-external-tool-internal-hash-is-fragile-binding`.) The thin-waist invariant therefore sharpens from "one byte-identical file across all harnesses" (a two-harness coincidence — both read markdown) to **"one shared source, projected per-harness: symlink when formats match, compile when they don't."**
  - **Revised 2026-06-06 (hook event-axis projection subset; manifest = documentation parity):** the shared hooks-manifest is the **documentation-parity source for CC hook events** — the set of `ccEvent` names in the manifest equals the set of CC hook-event names configured in `settings.json` that carry ≥1 concrete hook command (an empty-array event like `PostToolUse: []` carries zero hooks and is excluded; the same ≥1-hook rule applies symmetrically to both sides, and adds an event automatically once CC configures a real hook for it). But **per-harness PROJECTION is a SUBSET along the event axis**, not just along the decision-surface axis already documented above: Codex's hook schema is `[[hooks.PreToolUse]]`-only (see the 2026-06-05 revision), so the lifecycle events (`SessionStart` / `SessionEnd` / `PreCompact` / `Stop` / `PermissionRequest`) **do not project to Codex at all** — their absence from the Codex config is correct, not a gap. pi receives only events that have a pi analog: the manifest's `piEvent` field is therefore **OPTIONAL** (a CC-only event carries no `piEvent`), and the pi runner must skip a `piEvent`-less entry rather than error on it. So the manifest documents the full CC event set, while each harness's installed projection is the subset that harness can consume. This is wiring, not a logic fork (D2's invariant holds): the projection step selects supported events per harness from one shared manifest; no per-harness hook *logic* diverges.
  - **Revised 2026-06-06 (instructions axis + import-pointer projection):** instructions are now a *worked* shared-asset axis, not just a named one. The canonical project-instructions source is the **repo-root `AGENTS.md`**, read **natively** by Codex and pi (a cwd candidate — same physical bytes, zero projection). Claude Code does not read `AGENTS.md`, so it binds via a **committed repo-root `CLAUDE.md` whose body is the single native import directive `@AGENTS.md`** (CC resolves the import relative to `CLAUDE.md` → the sibling `AGENTS.md`). This is a **third projection mechanism** beside symlink and compile: a **static committed import-pointer** — used when one harness needs a one-line indirection to a file the others read natively, and the pointer is content-independent (a fixed one-liner regardless of `AGENTS.md` content), so it is **committed, not emitted by the D6 compile-skill, and carries no managed-block/idempotency machinery** (deliberately avoiding an accumulation bug found empirically — there is no emission to make idempotent). The projection-mechanism enumeration sharpens to **symlink | compile | import-pointer**. Scope boundaries are load-bearing: the **global-pi** instructions asset (`dotpi/agent/AGENTS.md`, symlinked to `~/.pi/agent/AGENTS.md` by install.sh) and the **personal global** agent instructions file (in the methodology home) are **separate axes, out of scope** — distinct in scope (global vs project) from the project-instructions axis, and they never competed. D2's no-fork invariant holds: a static committed pointer is wiring, carrying zero runner logic. Proven **live** (live round-trip verified): a non-`--bare` `claude -p` at the repo root echoes a sentinel that exists only in repo-root `AGENTS.md`, with a chain-less-cwd **negative control** confirming the import is the causal path (not ambient context). **Caveat:** CC `--bare` skips `CLAUDE.md` auto-discovery and is slated to become the `claude -p` default — a `--bare` spawn must pass instructions explicitly (e.g. `--append-system-prompt-file AGENTS.md`) or it silently drops them.
- **Rationale:** The substrate's portable half (instructions/skills) already has open standards; the discipline half (hooks/subagents) diverges structurally per harness and cannot be copied. The thin-waist keeps one source of truth for *logic* while accepting that *wiring* differs. Proven precedent in the existing setup: the `dcg` destructive-command-guard runs in **both** harnesses from **one binary** (CC `PreToolUse(Bash)` JSON hook ↔ pi `dcg.ts` `pi.on("tool_call")` extension).
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Single source compiled to each harness's native format (full behavior-generator) | Maximally DRY | **Rejected** — reasoned: the logic already lives in shared scripts/role-files, so there is little duplication left to eliminate; a behavior code-generator is cost without payoff. |
  | Native per-harness + a parity-checker | Lightest to build | **Rejected** — reasoned: reintroduces the hand-maintained duplication (14 roles ×2, hook table ×2) that the thin-waist removes. |
  | Adopt an off-the-shelf compiler (e.g. Ruler) | Exists today, supports pi | **Rejected (for now)** — external: Ruler-class tools carry instructions/skills/MCP well but the discipline-half (hooks/subagents) is exactly what they don't carry; reasoned: a config-mapping layer also cuts against the "keep it simple" goal. Re-evaluate if hand-rolled compile-skill grows unwieldy. |
- **What would invalidate this:** a discipline-half element resists becoming a shared CLI/asset and forces genuine per-harness logic duplication (not just trigger-wiring) — at which point single-source-compile or native+parity becomes the better trade. **Sharpened 2026-06-04 (hooks):** the concrete trigger for hooks is the protocol-adapter enum growing *unbounded*, or a script needing *bespoke per-script branching in the runner* (rather than a new entry in the fixed adapter map) — that is genuine per-harness logic. A fixed enum that occasionally gains one new protocol-class entry is not the trigger; per-script special-casing is. **Sharpened again 2026-06-04 (Codex spike):** a per-harness *compile* step projecting the one shared source into a harness's native format (e.g. Codex's TOML role file) is wiring, not the trigger — the invalidation is divergent per-harness *logic/behavior*, not divergent *file format*.

### D3: Shared role-brief library + per-harness spawner (a role registry)

- **Firmness:** EXPLORATORY (revised 2026-06-04)
- **Decision:** Subagent roles are **named files in a canonical shared directory** (`dotpi/agent/roles/<name>.md`, symlinked to `~/.pi/agent/roles/`). The **shared asset is the role BRIEF + neutral metadata** — `name`, `description`, `tools` (primitive allowlist), `skills` (SKILL.md folders to load — a **separate axis** from tools), `output-contract` (optional), and the body (harness-neutral system prompt). Claude Code reads these as `$CLAUDE_HOME/agents/*.md` (it already does). **pi's `subagent` extension is evolved to load a named role** and expand it: read+inline the named SKILL.md bodies, set `model`/`tools`/`thinking`, derive `expectedFinalLine` from `output-contract`, prepend the role brief to the caller's specific task.
  - **Revised 2026-06-04 (Claude Code schema spike) — the `model` field is per-harness, not shared.** The spike established that Claude Code's agent `model:` frontmatter accepts only Anthropic keywords/model-IDs (`sonnet`, `claude-...`, `inherit`), **never a `provider/model` slug**, and CC **cannot per-role route a subagent to a non-Anthropic provider** at all (its only provider-redirect is the global `ANTHROPIC_BASE_URL`, not a per-role pin). Therefore: CC reads `model:` (an Anthropic keyword/ID/`inherit`) and runs the role on Anthropic; **pi reads its explicit provider/model slug (D4) from a DISTINCT frontmatter field (e.g. `pi-model:`) that CC ignores**, keeping the file single + byte-identical (CC silently ignores unknown frontmatter). **Per-role cross-provider routing is a pi-only capability** — Claude Code shares the role's *discipline* (brief / skills / tools / output-contract), not its provider. The byte-identity thin-waist (D2) thus applies to the brief + neutral metadata; the model line is the one per-harness-wiring field, carried in the same shared file.
  - **Revised 2026-06-04 (Codex consumption-model spike) — "byte-identical file" is a two-harness property; the three-harness invariant is "one shared source, projected per-harness."** Claude Code and pi both read markdown role files, so one byte-identical file (pi's slug in a CC-ignored frontmatter key) serves both. **Codex role definitions are TOML** — categorically not the same physical file — so Codex binds from the single shared source by **generating its role file at compile-time** (the D6 compile-skill), not by symlinking. Single source is preserved (no fork); only the *projection mechanism* differs (symlink for CC/pi, compile for Codex). On the model field: **Codex splits provider and model into two separate fields and speaks only the OpenAI *Responses* wire protocol** (`WireApi::Responses`; the chat-completions wire is removed) — so Codex's analog of the per-harness model line is two fields. Per-role routing of Codex *itself* to a non-OpenAI provider is **available only via a provider that exposes an OpenAI-Responses-compatible endpoint** (a custom `model_providers` entry with such a `base_url`); plain non-OpenAI providers — Anthropic-native, vanilla OpenRouter — are not reachable without such a shim, unlike pi, which routes natively across wire formats. This is immaterial to the three-harness round-trip, which runs Codex natively on OpenAI; whether the reference spike provider (openrouter) exposes a Responses-compatible endpoint for Codex is unverified and off the round-trip's critical path. The per-harness-model-field principle (CC keyword / pi slug / Codex two-field) holds across all three; only its surface form — and, for Codex, its reachable-provider set — differs. **Concrete field name (determined during Codex spike follow-on work):** the shared role brief carries a third per-harness model field **`codex-model:`** (parallel to `pi-model:`), holding a `provider/model` slug (e.g. `openai/gpt-5.5`). The D6 compile-emitter splits it into the Codex TOML **`model`** (bare model id, e.g. `gpt-5.5`) and **`model_provider`** (the `model_providers` table key, e.g. `openai`). CC ignores `codex-model:` as unknown frontmatter and pi ignores it too (pi reads only `pi-model:`), so the markdown brief stays byte-identical-serviceable for CC+pi — the same CC-ignored-frontmatter mechanism that carries `pi-model:`.
  - **Revised 2026-06-06 (role-class field optionality):** field requirements are **role-class-dependent**, not uniform across all roles. Two classes: **reference roles** (the haiku-tier extraction agents like `file-scanner` / `content-extractor`) genuinely carry a resolvable `skills` ref and a terminal `output-contract`, and for them both fields are **REQUIRED**. **General-purpose roles** (the ~12 ported judgment agents — `implementer`, `adversarial-reviewer`, `debugger`, etc.) mostly reference skills that live in the methodology home (e.g. `adversarial-review`, `harness`, the `superpowers:*` namespace) **not** in the shared `agent/skills/` folder, and most have **no natural terminal output-contract**. For them `skills` and `output-contract` are **OPTIONAL — omittable with an explicit in-frontmatter rationale**. The validator must distinguish **deliberate omission (carrying a rationale) → PASS** from **accidental omission (no rationale) → FAIL**; a silently-omitted field is a broken role, an omitted-with-rationale field is a legitimate general-purpose role. (This also resolves a prior drift: D3 already listed `output-contract` as "(optional)" while `validate_roles.py` hard-required it for every role; the role-class split is the principled reconciliation — optional for general-purpose, required for reference.) The concrete frontmatter syntax (a `role-class:` marker, or a per-field `omit`+rationale pair) is an implementation choice carried in `validate_roles.py`; the load-bearing decision is the deliberate-vs-accidental distinction, not the syntax.
- **Rationale:** What is genuinely shared between a CC subagent and a pi subagent is the *role brief* (harness-neutral prose) plus neutral metadata; what differs is only the spawn mechanism. One shared role-definition library, consumed by each harness's native spawner, is the thin-waist applied to subagents. `tools` and `skills` are distinct because tools are primitive capabilities (`read`, `bash`) while skills are SKILL.md procedural-knowledge folders (which pi children load by inlining; grounded in pi's `_expandSkillCommand`).
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Keep pi subagents inline-brief-only (ADR-001 D2 original) | MVP simplicity | **Rejected** — direct: ADR-001 D2's own invalidation ("inline briefs become repetitive enough that prompt drift dominates") is now met by the cross-harness goal; the role-loader is additive (inline still works). |
  | Merge `tools` and `skills` into one field | Fewer fields | **Rejected** — reasoned: they are orthogonal axes (primitive capability vs procedural-knowledge folder); conflating them loses the skill-load semantics pi and CC both need. |
- **What would invalidate this:** the role metadata cannot be expressed compatibly across both spawners (e.g., a field one harness needs that the other actively rejects), forcing per-harness role files after all. **Partially triggered + resolved 2026-06-04 (schema spike):** the `model` field IS expressed differently per harness (CC: Anthropic keyword/ID in `model:`; pi: provider/model slug in a CC-ignored `pi-model:` field) — but this did **not** force per-harness FILES, because CC silently ignores unknown frontmatter, so one byte-identical file carries both and the brief stays fully shared. Full invalidation would require a field one harness *rejects the whole file over* — which is precisely why pi's slug lives in a separate CC-ignored field rather than overloading `model:` (CC's behavior on an unrecognized `model:` value is undocumented and was treated as unsafe to rely on). **Extended 2026-06-04 (Codex spike):** a harness needing its role file in a *different format* (Codex TOML) is NOT the trigger — compile-time projection from the one shared source preserves single-source. The trigger remains a role *field/content* element one harness needs that another rejects such that one shared source cannot express it at all.

### D4: Explicit provider/model slugs, not a modelTier abstraction

- **Firmness:** EXPLORATORY
- **Decision:** Roles pin **explicit provider-qualified model slugs** — `anthropic/claude-sonnet-4.5`, `openai/gpt-5.5`, `moonshotai/kimi-k2.6`, `openrouter/<vendor>/<model>` — rather than a `basic/medium/smart/max` tier. **Supersedes dotpi ADR-001 D3** (modelTier), which is evolved in place. Per D3's 2026-06-04 revision, this slug is carried in the pi-specific `pi-model:` frontmatter field (Claude Code's `model:` cannot hold a slug). Setup precondition: each pinned provider needs credentials/auth in `~/.pi/agent/auth.json` (else the child errors at spawn). Auth mechanism varies by provider and is a *separate concern* from the slug: native-API-key providers (anthropic-key, openrouter) resolve via `auth.json`/env (incl. `$ENV_VAR` interpolation); subscription providers (openai-codex, anthropic Pro/Max) require an interactive OAuth `/login`; non-built-in gateways (e.g. requesty) require a `~/.pi/agent/models.json` custom-provider block. A slug alone is necessary but not sufficient.
- **Rationale:** pi's `--model` resolves `provider/model` slugs verbatim against a built-in catalog, independent of the parent's default provider (grounded in `model-resolver.js:266`), so a slug "just works" and makes per-role multi-provider routing explicit and honest. The original modelTier rationale ("don't bake provider IDs into reusable skills") is outweighed once the substrate is deliberately multi-provider and per-role routing *is* the feature.
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Keep `modelTier` (ADR-001 D3) | Provider-agnostic intent | **Rejected** — reasoned: the abstraction hides the multi-provider routing that is now a first-class goal; explicit slugs are clearer and pi resolves them natively. Tier can return as optional sugar later. |
  | Bare model names (no provider prefix) | Less verbose | **Rejected** — reasoned: ambiguous across providers; the `provider/` prefix is the unambiguous canonical form pi's resolver prefers. |
- **What would invalidate this:** maintaining explicit slugs across many roles becomes a migration burden when models churn, and a native pi alias system solves capability-tier mapping better (this is ADR-001 D3's original invalidation, inherited).

### D5: CLI-first / no-MCP as the cross-harness tool stance

- **Firmness:** FLEXIBLE
- **Decision:** Tools are authored as **CLIs with READMEs** (invoked via `bash`), not MCP servers, as the cross-harness default — adopting pi's deliberate no-MCP stance. Existing Claude-Code MCP tools either get CLI equivalents or remain CC-only; new tools are authored as CLIs.
- **Rationale:** pi structurally rejects MCP (token-bloat stance); CLIs-with-READMEs are the portable tool shape both harnesses invoke identically via `bash`, and they fit progressive-disclosure. MCP is the portable layer *between CC and Codex*, but pi structurally rejects it — so CLI-first remains the cross-harness lowest-common-denominator even with Codex now first-class (revised 2026-06-04); whether Codex's own MCP support warrants Codex-only MCP tools is a question the Codex spike (D1) informs.
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | MCP as the cross-harness tool layer | It is the broad cross-tool standard | **Rejected** — direct: pi does not support MCP by design; reasoned: with Codex deprioritized, MCP buys nothing pi can use. |
  | Dual: MCP for CC, CLI for pi | Use each harness's native strength | **Rejected (as default)** — reasoned: doubles tool maintenance; CLI-first gives one shape both run. CC-only MCP remains allowed for tools that can't be CLI-shaped. |
- **What would invalidate this:** a critical tool genuinely cannot be CLI-shaped (e.g., needs persistent bidirectional session state MCP provides), making CC-only MCP unavoidable for that tool.

### D6: Compile/install is an agent-run skill, not a static compiler

- **Firmness:** EXPLORATORY
- **Decision:** Compilation/distribution of the substrate to a target harness is an **agent-run skill** — mechanical scripts/CLIs for the deterministic 80% (symlink shared assets, generate the `settings.json` hooks block from the shared manifest, place role files) plus SKILL.md judgment for the messy 20% (edge cases, model-field translation, new-harness quirks) — **not** a brittle static compiler that must enumerate every case.
- **Rationale:** The agent fills edge-case gaps with judgment instead of the author pre-enumerating them, which dissolves the power-vs-simplicity tension of a full compiler. This matches where the field landed: the `install.md` "agent reads instructions and sets itself up" standard (late 2025) folded into `skill.md` by Jan 2026 — skills carry both install and usage knowledge.
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Static deterministic compiler/installer | Reproducible, idempotent | **Rejected (as primary)** — reasoned: must cover every edge case or break; the user's explicit lean is agent-figures-it-out simplicity. Deterministic scripts remain the *substrate* the skill calls. |
  | Pure natural-language install instructions (no scripts) | Maximum simplicity | **Rejected** — external: pure agent-driven install is the highest-RCE-risk surface (2025–26 CVEs; "your LLM is not a security boundary"); needs the deterministic, sandboxed underlayer. |
- **What would invalidate this:** the compile surface needs more determinism/reproducibility than an agent-run skill provides (e.g., drift between machines becomes a real problem), pushing toward a pinned static installer. **Security note:** agent-driven install must run inside a container sandbox (established by the containerized-agent safety project's ADRs governing the sandbox) — the sandbox is the mitigation for the RCE surface this decision opens.

### D7: Open-source distribution posture — public core + private overlay, one-way reference

- **Firmness:** EXPLORATORY
- **Decision:** The substrate is split into a **public core** and a **private overlay**, distributed as a reusable open-source repo others install and adapt.
  - **Public core** (a new dedicated public repo): the generic agentic-engineering substrate — the workflow methodology skills (`recall`, `decompose`, `adversarial-review`, `compound`, `scout-adrs`, `scope-check`, `harness`, `adr-write`, `send-it`, …), the cross-harness mechanism (subagent role-loader, hook-runner, compile/install skill, manifests), and the **meta-ADRs that govern them** (this ADR and its siblings), **scrubbed of private evidence**. The core carries its **own ADR / bead / skill namespaces**.
  - **Private overlay** (dotpi + a private containerized-agent safety project + work repos): personal/work/product config, secrets, project beads, product-specific ADRs, the personal agent instructions file.
  - **One-way reference rule (the keystone):** the private overlay may reference the public core; the public core **never** references anything private. This makes the public core *publishable by construction* — nothing private can leak through a reference.
  - **Full-stack, no bundled binaries:** the core's tool dependencies (beads, CASSMS) are **named and shared**, provisioned by a **flywheel-style dependency manifest** that installs them from upstream and applies the core's configuration (with doctor/verify checks), rather than vendoring binaries.
  - **Scrub obligation:** core ADRs that currently cite private *evidence* (specific private-project dogfooding — e.g. a private containerized-agent safety project, product beads) anonymize the evidence while keeping the decision + warrant; skills that cite private *bead IDs* as spec-source sever or re-home those citations. Both are one-way leaks (core → private) the keystone rule forbids. A committed skills-canonicalization inventory (51 skill folders bucketed neutral / cc-specific / pi-specific / needs-adaptation) is the first cut of the boundary map; it extends to hooks, roles, and instructions.

  **Operational decisions (added 2026-06-07 — W3 framing pass; all EXPLORATORY, expected to move).** D7 above fixed the *posture* (public core + private overlay + one-way rule); these fix the *mechanics* the posture left abstract. Each carries its own invalidation; D7's three top-level invalidation predicates (one-way unmaintainable / scrub guts warrant / no adopter materializes) continue to govern the whole.

  - **D7.1 — Distribution model = canonical-home move-out** (sharpens "a new dedicated public repo"). The public core repo is the **new canonical home** of both the methodology (workflow skills + the meta-ADRs that govern them, today in the methodology home) and the cross-harness mechanism (today in `dotpi/agent/` + dotpi ADRs 001–003). These assets **physically move** into the public repo; the methodology home and dotpi then **install/symlink *from* it** via the D6 compile/install skill. One source of truth, public-by-construction (no separate scrub-on-publish step). *Alternatives:* (a) generated scrubbed export/mirror — **rejected**, reasoned: two-place maintenance + permanent export drift + source-of-truth ambiguity; (b) staged export-then-flip-to-move-out — **rejected**, reasoned: more machinery for a confidence-hedge the user chose not to take; commit to the end-state directly. *Invalidation (signal-shaped):* the methodology home/dotpi accumulate local overrides diverging from the published core often enough that "install-from" stops being a faithful consume — signalling the core isn't actually harness-neutral / the move-out boundary is drawn wrong.

  - **D7.2 — Repo identity + namespace reconciliation.** The public core repo is **`github.com/jsnyde0/harness`**. Core bead namespace prefix = **`harness-`** (via beads' existing directory-name auto-detect — no new mechanism). ADR namespace: the mechanism ADRs (dotpi **001–003**) and the methodology ADRs (methodology home **004–014**) occupy **disjoint number ranges**, so they coexist in the public repo's single ADR namespace with **no renumber**. The methodology home's ADR-001/002/003 (a private approvals project, containerized-agent-config, account-switch) are **private** and stay in the overlay (their core-vs-private content classification is to be **verified at execution**, not assumed here). *Invalidation:* a future core ADR needs a number already held by a private methodology-home ADR (range disjointness breaks) → forces an explicit renumber or namespace-prefix scheme.

  - **D7.3 — Boundary representation = central manifest, private-by-default.** A single **core-manifest** file enumerates the core asset paths; **everything not listed is private by default and does not publish**. This is the machine-readable boundary that supersedes the prose skills-canonicalization-inventory (the inventory remains the human-readable first-cut map). Classification extends beyond skills to **hooks, roles, instructions, ADRs**. *Rationale:* both the move-out (D7.1) and the automated one-way check (D7.4) need a machine-knowable core set; private-by-default **fails safe** — a forgotten core asset stays private (under-publishes) rather than leaking. *Alternatives:* (a) per-asset frontmatter `classification:` tag — **rejected**, reasoned: more robust/self-describing but heavy upfront tagging across ~51 skills + hooks/roles/ADRs; user chose single-source auditability; (b) prose inventory only — **rejected**, reasoned: not machine-readable, cannot drive an automated move or one-way check. *Invalidation:* the manifest drifts from reality often enough (core assets repeatedly forgotten, or the published set repeatedly wrong) that per-asset tagging would have been the safer representation.

  - **D7.4 — One-way-reference check = script engine + pre-push hook + CI, with a mandatory planted-leak self-test.** One reusable **check-script engine** is invoked by **both** a local git **pre-push hook** and public-repo **CI** (defense-in-depth, single source of check logic). It scans the manifest-listed core assets for private markers: private paths (a private containerized-agent safety project, work repos, hardcoded methodology-home paths, **work-org directory names**), work GitHub accounts, private bead-ID cites — **including the methodology's OWN bead-history namespace, not just the private project namespaces** (resolved 2026-06-08 during boundary correction: per D7.2 the public core gets a fresh `harness-` bead namespace, so the original methodology bead store stays in the private overlay and cites to it are severed / faithful-descriptor'd per D7.5; the earlier carve-out treating methodology-own bead history as "travels with the methodology" is **rejected** — direct: a public asset citing a private bead store is a one-way leak the keystone rule forbids, the same as any other private-bead cite). **Non-negotiable:** the check ships with a **planted-leak self-test fixture it provably FIRES on** — guarding the verification-artifact anti-pattern (a check that exits clean but never actually catches a real leak; cf. bd memory `deny-path-hook-test-use-harmless-sentinel`). **Marker externalization (resolved 2026-06-08 during the W3 pre-publish scrub):** the author's real private markers (work-org tokens, private-project nouns, private bead-ID prefixes, private source paths) are **not hardcoded in the committed engine** — they live in a **gitignored local config** the engine reads, alongside a committed **generic example template** an adopter copies and populates with their own. This makes "reusable engine" literal: the published detector ships **zero real private nouns** yet still fires for any adopter who supplies markers, and it closes the self-leak the hardcoded list would otherwise be — the detector was itself the last reservoir of the very nouns the rest of the scrub removed (necessary-not-sufficient: a green leak-check does not catch its own marker list shipping). Structural classes (literal methodology-home and absolute macOS home-path *shapes*, which carry no identity) stay hardcoded; only author-identifying tokens externalize. The self-test fixture uses **synthetic markers**, proving fire/no-fire discrimination (bare provenance word passes, bead-ID shape fires) without embedding real nouns. *Alternatives:* (a) CI only — **rejected**, reasoned: first feedback only post-push (already on GitHub); (b) local hook only — **rejected**, reasoned: bypassable via `--no-verify`, unenforced for outside contributors; (c) hardcoded marker list in the committed engine — **rejected** 2026-06-08, direct: the detector then publishes the exact private inventory it exists to protect, and is not reusable by an adopter with different private nouns. *Invalidation:* the marker set proves incomplete (a real leak ships past a green check) OR keeping hook+CI parity becomes its own divergence source OR an adopter cannot run the check on a fresh clone because the example-template fallback is missing keys the engine reads.

  - **D7.5 — Scrub criterion = uniform faithful-descriptor.** When scrubbing private evidence from core ADRs, apply one mechanical rule everywhere: replace private proper nouns / bead-IDs / paths with **anonymized-but-faithful descriptors** (e.g. "a private containerized-agent safety project" for a private containerized project name), keeping the citation/warrant *structure* intact. Operationalizes D7's scrub obligation (keep decision + warrant). *Alternatives:* (a) case-by-case (default descriptor, but rewrite the warrant self-standing where a citation is load-bearing) — **rejected**, reasoned: most fidelity but heavy per-ADR judgment; user chose the uniform scriptable rule; (b) generalize all warrants self-standing, dropping citations entirely — **rejected**, reasoned: over-corrects past D7's predicate 2 — the dogfooding "battle-tested" force *was* load-bearing. *Invalidation (this IS D7's top-level predicate 2, made concrete):* uniform descriptor-substitution softens a specific warrant enough that the meta-ADR stops being credible → that ADR needs the case-by-case self-standing rewrite instead.

  - **D7.6 — Portable-form convention for core path references (added 2026-06-07 during the W3 scrub-bead execution; EXPLORATORY).** D7.4 says which path forms are *forbidden* in core (the C2 marker: literal tilde-home and absolute-home `.claude` references); this says which to use *instead*, per execution context: (a) **`$CLAUDE_HOME/...`** in the compiled hooks-manifest — a template the **consumer resolves** to the target's methodology home (`install.sh`'s `expand_home` for the Codex-TOML projection; the pi runner's `resolveScriptPath` for runtime), NOT assumed present in the shell environment; (b) **`$SKILL/...`** for a skill referencing its *own* files (set per-harness at skill-invocation); (c) **`$HOME/.claude/...`** for ad-hoc shell commands in skill/role bodies that an agent runs directly (shell-expands in any shell; not a C2 marker). *Rationale:* the leak-check (D7.4) is necessary-not-sufficient — a path form must *also resolve in its execution context*. Without a stated convention, the first W3 scrub blindly substituted an **undefined `$CLAUDE_HOME`** everywhere; it passed the leak-check *hollowly* (not a C2 marker) while breaking the security-hook wiring (resolved to empty in pi runtime, Codex TOML, and agent shells). *Alternatives:* (a) uniform `$CLAUDE_HOME` everywhere incl. shell-run commands — **rejected**, direct: unset in agent/hook/compile shells, so shell commands break (the W3 scrub regression); (b) keep literal tilde-home `.claude` paths (tilde resolves) — **rejected**, reasoned: trips the C2 marker / harness-couples the public core. *Invalidation:* a core asset needs a path reference none of {`$CLAUDE_HOME`-compiled, `$SKILL`, `$HOME/.claude`} can express portably (taxonomy incomplete → extend it); OR a new consumer of the manifest is added without wiring `$CLAUDE_HOME` resolution (the token silently breaks again — the scrub's harness must therefore assert *mechanism-intact*, not just *leak-gone*).

- **Rationale:** The thin-waist (D2) and agent-run compile/install (D6) already presuppose a portable, manifest-driven substrate; D7 takes that to its logical end — portability *between the author and an open-source community*, not just between CC and pi. The one-way rule is the minimal invariant that makes "publishable" mechanical rather than a per-commit audit. **Broad core** (methodology + mechanism + meta-ADRs) over engine-only because the methodology *is* the substrate's value; withholding it ships plumbing without the discipline that makes the plumbing worth installing.
- **Alternatives:**

  | Option | Why considered | Verdict |
  |---|---|---|
  | Private-is-home, publish via a scrub pipeline downstream | Zero daily friction; keep personal refs freely | **Rejected** — reasoned: a scrub transform is fragile and bidirectional drift is permanent; the author never dogfoods the public artifact, and forkers can't contribute back (their home ≠ the author's). |
  | Engine-only / narrow core (mechanism + examples; methodology + ADRs stay private) | Smallest scrub surface, ships fastest | **Rejected** — reasoned: withholds the most valuable part (the workflow methodology); "adapt" degrades to "build your own," a high adoption bar that defeats the reusable-substrate goal. |
  | Bundle the tool binaries (beads, CASSMS) into the repo | Single-clone install | **Rejected** — external: the `agentic_coding_flywheel_setup` prior art and standard packaging both manage external tools via upstream-fetch + SHA256 checksum, not vendoring; reasoned: vendoring binaries bloats the repo, breaks cross-platform, and forks upstream maintenance. |
  | Keep everything in one private repo (status quo) | No extraction cost | **Rejected** — direct: the user's goal is a publicly shareable reusable substrate; sharing all of dotpi would expose private project/work content (paths from a private containerized-agent safety project, work GitHub accounts in `CLAUDE.md`). |
- **What would invalidate this:** the one-way rule proves unmaintainable in practice (the core keeps wanting to reference private context — a signal the core/overlay boundary is drawn in the wrong place); OR scrubbing private evidence guts the meta-ADRs' dogfooding warrant enough that they stop being credible (the evidence *was* load-bearing, not decoration); OR no external adopter ever materializes and the public/private split is pure overhead with no offsetting reuse — at which point a single private repo (status quo) is the better trade.

## Consequences

- pi's `subagent` extension gains a role-loader (additive; inline still works) — see ADR-001 D2 evolution.
- A shared `agent/roles/` library and a shared hooks-manifest become the canonical discipline-half source; both harnesses' wiring is generated/symlinked from them.
- A `compile`/install skill is the distribution mechanism; it must run sandboxed.
- Each provider a role routes to needs credentials in `~/.pi/agent/auth.json`.
- Codex is a **first-class target** (revised 2026-06-04); a consumption-model spike precedes its wiring. MCP tools become CC/Codex-only legacy unless ported to CLIs (pi cannot consume MCP).
- The substrate is extracted into a **public core repo + private overlay** (D7); the compile/install skill (D6) grows a dependency-manifest provisioning responsibility (beads + CASSMS from upstream, no bundled binaries).
